# LIFE OS 画面圧縮・入力負荷軽減 実装設計指示書

## 1. 目的

現在の `LIFE OS` は機能は揃っているが、スマホではスクロール量が多く、入力前の心理的ハードルが高い。

今回の改修では次の 4 点を同時に達成する。

1. `Today` の名称を `Morning` に変更する
2. `今週の目標` を短文前提の小さな入力に変える
3. `タスク` 画面で朝に決めた `今日の最重要3つ` を常時見える位置に置く
4. `連絡 / 121日程調整 / 作業タスク` を折りたたみ式にして、スクロール量を大きく減らす

加えて、全体の冗長さを減らすための改善案を実装判断しやすい形で提案する。

## 3. Morning 画面の変更

## 3.1 ページ名

`Today` のページ名はすべて `Morning` に変更する。

対象:

- 下部ナビ `Today` -> `Morning`
- 画面キッカー `Morning Board` は維持してよい
- ユーザー向け説明や最終案内でも `Today` ではなく `Morning` と呼ぶ

内部実装:

- `data-screen="today"` や `state.activeScreen = "today"` は変更不要
- ナビラベルや見出しだけ変更する

## 3.2 今週の目標

`今週の目標` は長文入力前提をやめ、1〜2 行の短文にする。

実装方針:

- `textarea rows="6"` を縮小する
- 推奨は次のどちらか
  - `textarea rows="2"`
  - `input type="text"` を 2 本にする

今回の推奨:

- `textarea rows="2"` を採用
- 理由:
  - 既存の保存ロジックをほぼ変えずに済む
  - 改行 1 回までは許容できる
  - UI の高さだけ下げられる

プレースホルダ例:

`今週のテーマ / 最優先KPIを短く書く`

## 4. タスク画面の再構成

## 4.1 画面冒頭のヒーロー削除

`タスク` 画面上部の次の塊は丸ごと削除する。

- `Task`
- `タスクを漏らさず整理する`
- 説明文 `発生した連絡、121日程調整、タスクを...`

理由:

- 情報価値に対して縦の占有が大きい
- 毎日見る画面としては説明より即入力の方が重要

削除後は、画面を開いた瞬間に `最重要3つ` と `引き継ぎ` と `折りたたみセクション` が見える構成にする。

## 4.2 最重要3つを常時表示

朝に決めた `今日の最重要3つ` を `タスク` 画面の最上部に固定表示する。

表示仕様:

- 画面冒頭に `今日の最重要3つ` カードを置く
- 中身は `Morning` 画面の `mit1 / mit2 / mit3`
- ここでは編集可能にしてもよいが、まずは `参照用表示` を優先する

今回の推奨:

- `参照用表示 + 1タップでMorningへ戻れる導線` にする
- 理由:
  - `タスク` 画面で編集可能にすると責務が増えて散らかる
  - まずは「今日の軸を見失わない」役割を優先する

推奨 UI:

- 3 行のコンパクトカード
- 空欄時は `未設定`
- 必要なら `Morningで編集` の小ボタンを置く

## 4.3 引き継ぎ表示

現状の `前日からの引き継ぎ` は維持するが、圧縮表示にする。

変更点:

- 3 つの独立カードを、1 枚のサマリーカードにまとめる
- 例:
  - `連絡 0件`
  - `121日程調整 1件`
  - `作業タスク 2件`

目的:

- 「引き継ぎがあるか」を一瞬で把握させる
- 縦の高さを減らす

## 4.4 折りたたみセクション

`連絡 / 121日程調整 / 作業タスク` は、常時全展開しない。

各セクションは次の構成にする。

1. セクションヘッダー
2. 件数と未整理数
3. 開閉トグル
4. 開いたときだけ入力フォームと一覧を表示

### 4.4.1 初期状態

初期状態は次のようにする。

- `連絡`: 閉じる
- `121日程調整`: 閉じる
- `作業タスク`: 開く

理由:

- 最も頻繁に使うのは `作業タスク`
- `連絡` と `121` は必要時だけ開けばよい

### 4.4.2 例外ルール

未整理項目があるセクションは、初回表示時だけ自動で開いてよい。

優先ルール:

1. `open` 項目があるセクションは自動展開
2. `open` がなければデフォルト開閉状態に従う

### 4.4.3 ヘッダー表示

各ヘッダーには以下を出す。

- セクション名
- 総件数
- 未整理件数
- 開閉アイコン

表示例:

- `連絡 3件 / 未整理1`
- `121日程調整 0件`
- `作業タスク 4件 / 未整理2`

## 4.5 一覧優先レイアウト

セクションを開いたときの表示順は以下に統一する。

1. 既存項目一覧
2. 新規追加ボタン
3. 新規入力フォーム

ただし、さらに入力ハードルを下げるため、デフォルトではフォームも閉じてよい。

今回の推奨:

- セクション本体を開く
- その中で `+ 追加する` を押したときだけフォームを表示

これにより、一覧確認だけなら非常に短いスクロールで済む。

## 5. スクロール量を減らす改善案

以下は今回の要望に直接含まれるものではないが、冗長さを減らすうえで有効な改善案である。

## 5.1 採用推奨: Compact モード

`Morning` と `タスク` でカード余白を少し詰める。

内容:

- カード上下余白を 10% から 18% 減らす
- セクション見出しの下マージンを縮める
- フォーム入力欄のデフォルト高さを少し下げる

理由:

- 情報量は維持したまま、スマホの 1 画面あたりに見える量が増える

## 5.2 採用推奨: 入力は隠し、一覧を先に見る

今の構成は「フォーム -> 一覧」なので、追加しない日でも縦に長い。

変更案:

- 各タスクセクションは `一覧優先`
- `追加する` を押したときだけフォーム展開

効果:

- 何も追加しない日はほぼ一覧だけで終わる
- 入力を始めるときだけ高さが増える

## 5.3 採用推奨: Morning の Handoff を圧縮

`昨日からの申し送り` は 3 枚カードではなく、1 枚の縦リストにまとめてよい。

例:

- `今日のアクション: ...`
- `昨日のリマインド: ...`
- `未来から逆算した今日の一手: ...`

効果:

- `Morning` 画面の冒頭高さをかなり減らせる

## 5.4 任意: 最重要3つを 1 行要約表示

`Morning` 冒頭または `タスク` 冒頭で、3つをカード 1 枚ではなく 3 行の短いリストにする。

効果:

- 軸は見える
- 高さは最小化できる

## 5.5 任意: 夜ルーティンも折りたたむ

将来的には `Night` も同じ設計に寄せられる。

ただし今回はスコープ外とし、まず `タスク` 画面の圧縮を優先する。

## 6. 推奨採用セット

今回の実装では、次のセットを推奨する。

1. `Today` -> `Morning` に名称変更
2. `今週の目標` を `rows=2` に縮小
3. `タスク` 画面ヒーロー削除
4. `今日の最重要3つ` を `タスク` 画面冒頭に常時表示
5. `引き継ぎ` を 1 枚の圧縮カードにする
6. `連絡 / 121 / 作業タスク` を折りたたみ化
7. デフォルトは `作業タスク` のみ開く
8. セクション内は `一覧先出し + 追加フォーム後出し`

このセットが、最も少ない改修でスクロール量と心理的負荷を大きく減らせる。

## 7. データ設計

新しく必要になるのは UI 状態だけで、業務データ構造は基本維持する。

`state` に次の UI 状態を追加する。

```js
ui: {
  taskSections: {
    contact: false,
    schedule121: false,
    task: true
  },
  taskAddForms: {
    contact: false,
    schedule121: false,
    task: false
  }
}
```

方針:

- 開閉状態はセッション内 state のみでよい
- 永続化は不要
- ただし必要なら `localStorage` に保存してもよい

5. `連絡 / 121日程調整 / 作業タスク` が折りたたみ式になる
6. 初期表示では全セクションが開きっぱなしにならない
7. 未整理項目があるセクションは自動で開ける
8. `追加する` を押したときだけ入力フォームが開く
9. スマホで見たとき、以前より明らかにスクロール量が減る
10. 「入力しなければならない感じ」が減り、一覧確認だけでも使える構成になっている

## 10. 実装上の注意

- スクロール量削減では「情報を消す」のではなく「最初は隠し、必要時だけ出す」を原則にする
- `最重要3つ` は `Morning` の軸なので、`タスク` 画面でも見失わない位置に置く
- `タスク` 画面では説明文より一覧と入力の即時性を優先する
- 折りたたみはアニメーションしすぎず、素早く開閉できることを優先する
