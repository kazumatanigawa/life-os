## 目的

今回の設計指示書にある変更を、依存関係を崩さずに 1 回の実装で完了させる。

対象は以下。

- morningページ
- 10mタイムマシン
- Morning最終ページ
- Taskページ
- settings画面
- 作業タスク保存ロジック
- localStorage保存処理
- タイムライン表示処理

## 実装方針

今回の変更は UI の見た目だけではなく、`settings`、`time machine`、`task schedule` の保存構造に跨る。
そのため、実装順は必ず以下に固定する。

1. 保存基盤を先に整える
2. 10mタイムマシンの質問と記録構造を差し替える
3. morningページの文言と導線を整理する
4. 作業タスクの時間モデルを `durationMinutes` 基準へ統一する
5. Task ページのアコーディオン UI を仕上げる
6. 既存データ互換と全体検証を最後に通す

この順にしないと、UI だけ先に変えて保存形式が壊れる可能性がある。

### 4. 10mタイムマシンの記録保存

目的:
- 当日の未来イメージを年単位で再提示できるようにする

保存仕様:
- `timeMachineVisions` を localStorage に保存
- 既存の daily log 保存とは別に、履歴配列として保持

推奨データ構造:

```json
[
  {
    "createdDate": "2026-06-04",
    "targetDate": "2029-06-04",
    "answers": {
      "worksAndBusiness": "",
      "solvedProblems": "",
      "incomeStructure": "",
      "socialImpact": "",
      "coreValue": "",
      "appearanceAndExpression": "",
      "messageFromFuture": "",
      "todayMostImportantAction": ""
    }
  }
]
```

実装内容:
- 10mタイムマシン完了時に記録追加
- 同一 `createdDate` と同一 `targetDate` の上書きか追記かを決める

推奨:
- 最短実装では追記
- 同日複数回の許容で問題ない

確認条件:
- 完了時に記録が保存される
- ローカル保存に answers が入る

## リスクと対処

### リスク1: 旧データ互換でタイムライン表示が壊れる

対処:
- `getItemDurationMinutes()` を唯一の duration 取得関数として統一
- 表示側で `endTime` を直接参照しない

### リスク2: 10mタイムマシン完了条件が崩れる

対処:
- 既存の checkbox 完了判定を外し、完了ボタン押下時に保存処理を明示実行

### リスク3: future vision の履歴と daily log の責務が混ざる

対処:
- `daily log` と `timeMachineVisions` は分離保存
- 参照だけを 10mタイムマシン開始時に行う

## 実装時の絶対ルール

- 既存データ構造を壊す全面置換はしない
- 新規保存は `durationMinutes` 基準
- 旧 `endTime` は読み取り互換のみ維持
- UI 削除対象は CSS 非表示ではなく DOM / render / state から外す
- 表示文言の変更だけで済む箇所はロジックを触りすぎない
