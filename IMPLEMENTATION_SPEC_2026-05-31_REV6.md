# LIFE OS Task 画面再編・タイムライン追加 実装指示書

## 1. 目的

今回の改修では、`Task` 画面をさらに「その日の実務管理」に最適化する。

特に次の5点を同時に達成する。

1. `編集` ボタンを廃止し、操作体系を単純化する
2. `明日に回す` を選んだときに、`いつやる` をその場で更新し直せるようにする
3. `いつやる` が決まっている作業タスクを時系列順で見られる `タイムライン` を追加する
4. Task 画面の表示順を、実務導線に沿って再構成する
5. `支出メモ` を `メモ` に名称変更する

## 3. Task 画面の表示順を再構成

Task 画面のカード順は次の順番に固定する。

1. 前日からの引き継ぎ
2. 今日の最重要3つ
3. 連絡
4. 日程調整
5. 作業タスク
6. タイムライン
7. メモ
8. 夜ルーティンへ進む

理由:

- まず昨日から持ち越した意思決定を確認する
- 次に今日の軸を確認する
- その後に新規発生業務を整理する
- 最後に時系列の実行順を一覧で確認する

実装指示:

- `index.html` の `day-ops-screen` 内のカード順をこの順で組み替える
- 現在先頭にある `今日の最重要3つ` は、`前日からの引き継ぎ` の下へ移動する
- `メモ` は一番下に置く

## 4. 編集ボタンを削除

`作業タスク` の `編集` ボタンは削除する。

理由:

- ユーザー運用では、不要になったタスクは削除して再追加すれば十分
- 編集機能は UI と状態管理を複雑にする
- Task 画面はできるだけ即断即決できる構成に寄せるべき

実装指示:

- `renderDayOpsList()` から `task` 用の `編集` ボタンを削除
- `startDayOpsEdit()` など編集用 state / 関数が不要なら削除する
- `task-form-submit-button` の `保存する` 切り替えも不要にする
- `編集をやめる` ボタンも削除する

受け入れ条件:

- 作業タスクカードに `編集` が存在しない
- 編集 state が残らず、追加フォームは追加専用に戻っている

## 5. 「明日に回す」時の再設定導線

## 5.1 目的

`明日に回す` を押したとき、単に持ち越すだけではなく、`いつやる` を再設定し直せるようにする。

意図:

- 明日に回した時点で「明日のどこでやるか」まで軽く決めたい
- 持ち越しタスクの解像度を上げる

## 5.2 対象

- 主対象は `作業タスク`
- `連絡` と `日程調整` は従来どおりでもよい

## 5.3 挙動

`作業タスク` で `明日に回す` を押したとき:

1. status は `carryOver` に変わる
2. 直後に `いつやる` の再入力 UI を出す
3. 再設定後に保存する

今回の推奨 UI:

- カード内インライン編集
- `いつやる` だけを編集する小さなフィールドをその場に出す

理由:

- フル編集 UI を戻さずに済む
- 必要な再入力は1項目だけ
- モバイルで速い

## 5.4 実装ルール

- `task` type の item に対し、`carryOver` 遷移直後だけ `reschedulePending` のような一時 state を持ってよい
- 入力欄の初期値は現在の `when`
- 保存後にその値を item.when に反映

推奨 state:

- `state.ui.reschedulingTaskId`
- `state.ui.reschedulingTaskValue`

ボタン:

- `保存`
- `キャンセル`

キャンセル時:

- 既存の `when` を維持したまま閉じる

## 6. タイムライン追加

## 6.1 目的

1日の中で `いつやる` が決まっている作業タスクを、時系列順で一覧できる `タイムライン` を追加する。

意図:

- タスク整理と実行順の確認をつなげる
- 「やることはあるが、順番が見えていない」状態を減らす

## 6.2 表示対象

対象は `dayOps.tasks` のうち、`when` が空でない項目。

表示対象ステータス:

- `open`
- `carryOver`

非表示:

- `done`

## 6.3 並び順

`when` が時刻として解釈できるものを優先して時系列順に並べる。

並び順ルール:

1. `HH:MM` または `H:MM` と解釈できるものを先に並べる
2. 時刻化できるもの同士は早い順
3. 時刻化できない文字列はその後ろに並べる
4. 時刻化できないもの同士は入力順または更新順で維持

例:

- `09:00`
- `14:30`
- `営業後`
- `移動中`

## 6.4 表示内容

各タイムライン項目には以下を出す。

- `いつやる`
- タスク名
- ゴール
- status

推奨 UI:

- 左に時刻またはラベル
- 右に内容カード
- 縦ラインでつながるタイムライン表現

## 6.5 空状態

対象タスクがない場合は以下を表示する。

- `まだ時間指定のあるタスクはありません。`

## 7. メモ名称変更

`支出メモ` は `メモ` に変更する。

理由:

- 実際には支出だけでなく雑記にも使える
- より広い用途に耐える

対象:

- Task 画面のカードタイトル
- placeholder 文言は必要なら少し広げてもよい
- Markdown 出力の見出し
- Logs 内の表記

推奨 placeholder:

- `支出や気づいたことを雑にメモしておく`

1. Task 画面の最上部が `前日からの引き継ぎ` になっている
2. `今日の最重要3つ` はその下に配置されている
3. `編集` ボタンが存在しない
4. `明日に回す` を押したとき、`いつやる` を再設定する UI が出る
5. `タイムライン` が存在する
6. 時間指定のあるタスクが時系列順に並ぶ
7. `121日程調整` 表記が残っていない
8. `支出メモ` が `メモ` に変わっている
9. Markdown / Logs の表記も `メモ` と `日程調整` に統一されている

## 12. 推奨実装順

1. Task 画面のカード順を組み替える
2. 編集ボタンと編集 state を削除する
3. `明日に回す` 時の `when` 再設定 UI を追加する
4. `タイムライン` 描画ロジックを追加する
5. `メモ` 表記に統一する
6. CSS でタイムラインと再設定 UI を整える

この順なら、構造整理 -> 不要機能削除 -> 新機能追加 -> 見た目調整の順で手戻りが少ない。
